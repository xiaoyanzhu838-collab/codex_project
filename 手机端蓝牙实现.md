## 2. 当前蓝牙架构（手机侧）

当前是**手机做 GATT Client**，眼镜做 GATT Server。

### 2.1 控制通道（自定义服务）

`BleManager` 中定义了当前实际使用的 UUID：

- Service: `3f6d8b2e-7f61-4c85-b1d2-6e9c0f4a12d7`
- Notify Characteristic（眼镜->手机）: `a1c4e5f6-9b32-4d8a-8c7f-2b9d4e6f1a23`
- Write Characteristic（手机->眼镜）: `b7d3c1a9-5e24-4fa1-9d6b-3c8e2f5a7b41`
- CCCD: `00002902-0000-1000-8000-00805F9B34FB`

> 注意：`ble.md` 中写的是 `FFF1/FFF2`，与当前 `BleManager.kt` 不一致。以代码为准，建议后续统一文档。

### 2.2 标准电池通道（独立）

`BatteryGattClient` 使用标准服务：

- Battery Service `0x180F`
- Battery Level `0x2A19`
- Battery Power State `0x2A1A`

它是单独的一套连接与回调，不走 `BleManager`。

## 3. 最重要：连接建立基础流程

下面是当前代码中**控制通道**连接应有的完整链路（由 `BleManager` 实现）：

1. **初始化**
   - 调 `BleManager.init(context)`
   - 内部拿到 `BluetoothManager` / `BluetoothAdapter`

2. **发起连接**
   - 调 `BleManager.connect(device: BluetoothDevice)`
   - 内部先 `closeGatt()` 清旧连接
   - 然后 `device.connectGatt(context, autoConnect=false, gattCallback, TRANSPORT_LE)`

3. **连接状态回调**（`onConnectionStateChange`）
   - `STATE_CONNECTED`：
     - 标记 `connected=true`
     - Android 5+ 先 `requestMtu(517)`，再进行服务发现
   - `STATE_DISCONNECTED` 或失败：
     - 清理状态/写队列

4. **MTU 协商回调**（`onMtuChanged`）
   - 成功则更新 `currentMtu`
   - 然后调用 `discoverServices()`

5. **服务发现回调**（`onServicesDiscovered`）
   - `getService(SERVICE_UUID)`
   - 取 notify 特征 + `setCharacteristicNotification(true)` + 写 CCCD 开通知
   - 取 write 特征
   - 进入可收发状态，开始尝试发送队列

6. **发送数据**
   - 上层调用 `sendCtrl(BleMsg)`
   - 封装为外层 JSON（`action/cmd/pkg/data.text`）
   - 加自定义头部（1字节 type + 4字节 seq）
   - 入队并串行写（`writeInFlight + pendingWrites`）

7. **接收数据**
   - `onCharacteristicChanged`
   - 先走 `PacketAssembler` 按 seq 组包
   - 组包成功后按 UTF-8 转字符串并 JSON 解析成 `BleMsg`
   - 通过 `Listener.onCtrlReceive(...)` 回调给上层

## 4. 数据封装方式（手机侧）

### 4.1 业务到 BLE 的两层结构

上层业务通常构造业务 JSON（例如 `{"action":"get_status","timestamp":...}`），然后放入：

- `BleMsg(action = "text_input", pkg = 目标包名, data = Payload(text = 业务JSON字符串))`

`BleManager` 再包装一层线上的 outer JSON：

- `action`
- `cmd = "ack"`
- `pkg`
- `data.text`

### 4.2 分包头

`framePayload()` 添加固定头：

- `type`：1 字节，固定 `0x01`
- `seq`：4 字节，大端整型
- `body`：JSON UTF-8 字节

接收侧 `PacketAssembler` 用同样规则按 `seq` 拼包。

## 5. 连接相关的现状与风险点（重点）

### 5.1 MainActivity 当前未看到完整接线

`MainActivity` 中有扫描按钮逻辑，但通过反射调用：

- `startNativeScan()` 或 `startScan()`
- `stopNativeScan()` 或 `stopScan()`

而 `BleManager.kt` 当前并未实现这些扫描 API（至少该文件内没有）。这意味着：

- 扫描流程可能不可用（按钮触发后容易提示“未找到 API”）
- 也看不到“扫描结果 -> 选设备 -> BleManager.connect(device)”的明确闭环

### 5.2 Listener 链路未见显式注册

`BleManager` 提供了 `addListener/removeListener`，用于：

- 连接状态变更
- 收到通知消息
- 扫描状态/扫描结果

但在 `MainActivity` 已读代码中未看到明确的 `ble.addListener(...)` 注册与 `ble.init(...)` 调用，需确认是否在其他未读入口调用。

### 5.3 文档 UUID 与代码 UUID 不一致

`ble.md`（FFF1/FFF2）与 `BleManager.kt`（3f6d... 等）不一致，联调时容易出现“连上但服务找不到”的问题。

## 6. BatteryGattClient 的连接方式（基础）

`BatteryGattClient.connectAndRead(address)` 的连接链路是标准 GATT 读流程：

1. `BluetoothManager.adapter.getRemoteDevice(address)`
2. `device.connectGatt(..., TRANSPORT_LE)`
3. `onConnectionStateChange` 里 `discoverServices()`
4. 找 `0x180F` 服务
5. 读 `0x2A19` / `0x2A1A` 并在支持 notify 时开通知
6. 在 `onCharacteristicRead/onCharacteristicChanged` 处理电量与充电状态

适合“读标准特征”的子场景，不承担控制通道业务。

---

## 7. 建议的“建立连接”标准落地方式

若要让当前工程连接链路可稳定运行，建议统一为以下步骤：

1. App 启动或主页面初始化时执行 `BleManager.init(applicationContext)`。
2. 注册一个 `BleManager.Listener`：
   - 在 `onStateChanged` 更新 UI
   - 在 `onCtrlReceive` 处理眼镜回包
3. 明确实现扫描模块（推荐直接在 `BleManager` 内实现 `startScan/stopScan`）：
   - Android 12+ 先检查 `BLUETOOTH_SCAN/CONNECT`
   - 扫描回调里把 `BluetoothDevice` 上抛
4. 用户选中设备后调用 `BleManager.connect(device)`。
5. 仅在连接已就绪后发业务消息（或继续利用当前队列，保持幂等重试）。
6. 页面销毁/退出时关闭扫描、移除 listener，必要时断开连接。

---

## 8. 一句话结论

本工程蓝牙核心收发机制（GATT 连接、MTU、服务发现、通知、写队列、组包）在 `BleManager` 中已经具备；**最关键缺口在“扫描与连接入口接线”是否完整可用**，尤其是 `MainActivity` 通过反射调用的扫描 API 与 `BleManager` 当前实现不一致，建议优先补齐这一段闭环。
