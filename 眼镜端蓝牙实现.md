# 眼镜端蓝牙实现梳理（jove_glasses）

## 1. 结论先行（架构总览）

这个工程里的“蓝牙交互”核心是**BLE GATT Server（外设模式）**，不是手机常见的 GATT Client 模式：

- 眼镜端（本工程）充当 **BLE Peripheral + GATT Server**
- 手机端充当 **BLE Central + GATT Client**
- 手机先扫描到眼镜广播，再连上 GATT，随后通过特征值进行双向消息通信

关键实现类：

- `app/src/main/java/com/goolton/interaction/ble/BleGattServer.java`
- 业务接入入口：`app/src/main/java/com/goolton/interaction/MainActivity.java`

另外有 `AidlContentCenterBridge`，但当前 MainActivity 中已经“保留旧命名、内部切到自定义 BLE GATT Server”，即主要链路已走 `BleGattServer`。

---

## 2. 蓝牙相关代码位置

## 2.1 Manifest 权限与能力

`app/src/main/AndroidManifest.xml` 声明了 BLE 相关权限与特性：

- 传统权限：`BLUETOOTH`、`BLUETOOTH_ADMIN`
- Android 12+：`BLUETOOTH_CONNECT`、`BLUETOOTH_SCAN`、`BLUETOOTH_ADVERTISE`
- 硬件能力：`android.hardware.bluetooth_le`（`required=false`）

> 注意：MainActivity 的运行时权限申请主要覆盖相机/录音/定位/Wi‑Fi；蓝牙权限在当前代码里依赖系统环境与安装权限策略（设备是 system app 场景）。

## 2.2 运行时初始化入口

`MainActivity.onCreate()` 中调用：

1. `initContentCenterBridge()`
2. 实际内部执行 `initBleGattServer()`
3. `new BleGattServer(this)` 并设置消息回调
4. 调用 `bleGattServer.start()` 启动 GATT 服务和广播

销毁时在 `onDestroy()` 调用 `bleGattServer.stop()` 清理。

---

## 3. 建立连接（重点）

这里按“手机连接眼镜”的时序梳理。

## 3.1 眼镜端启动 BLE 服务

`BleGattServer.start()` 做了三件关键事：

1. 检查 `BluetoothAdapter` 是否可用且已启用
2. `startGattServer()`：创建并注册 GATT Service/Characteristic
3. `startAdvertising()`：开启 BLE 广播，允许手机扫描和连接

如果任何一步失败，会日志报错并返回 false。

## 3.2 GATT 服务定义（连接后交互基础）

`startGattServer()` 注册了一个主服务和两个特征：

- Service UUID：`3f6d8b2e-7f61-4c85-b1d2-6e9c0f4a12d7`
- Notify Characteristic（眼镜 -> 手机）：`a1c4e5f6-9b32-4d8a-8c7f-2b9d4e6f1a23`
  - `PROPERTY_NOTIFY`
  - 带 CCC 描述符 `0x2902`（客户端写入开启通知）
- Write Characteristic（手机 -> 眼镜）：`b7d3c1a9-5e24-4fa1-9d6b-3c8e2f5a7b41`
  - `PROPERTY_WRITE | PROPERTY_WRITE_NO_RESPONSE`

这两个特征构成了整个上行/下行消息通道。

## 3.3 广播与被发现

`startAdvertising()` 配置：

- 可连接：`setConnectable(true)`
- 广播包带 Service UUID
- 扫描响应包包含设备名（`setIncludeDeviceName(true)`）

手机端通过扫描 Service UUID 或设备名即可发现眼镜。

## 3.4 连接建立

当手机连接成功，回调 `onConnectionStateChange()`：

- `newState == STATE_CONNECTED`
  - 记录 `connectedDevice`
  - `notificationsEnabled = false`（初始默认不开通知）
- `newState == STATE_DISCONNECTED`
  - 清理连接设备、通知状态、分包缓存

> 也就是说，**“物理连接成功”不等于“可下行发消息”**，还需要下一步开启通知。

## 3.5 客户端开启通知（非常关键）

手机端需要对 Notify 特征的 CCC 描述符写入 `0x0100`。

服务端在 `onDescriptorWriteRequest()` 中识别该写入后：

- 设置 `notificationsEnabled = true`
- 后续 `sendBytes()/sendJson()` 才允许发送

如果没开通知，`sendBytes` 会直接日志提示并返回 false。

---

## 4. 连接后数据交互机制

## 4.1 手机 -> 眼镜（Write 通道）

入口回调：`onCharacteristicWriteRequest()`。

支持两种写法：

- 普通写（直接收到 value）
- Prepared Write（分段缓存，最终在 `onExecuteWrite()` 合并）

合并后的 payload 进入 `handleIncomingValue()`，继续 `decodeIncoming()`。

## 4.2 眼镜 -> 手机（Notify 通道）

调用链（MainActivity 侧）：

- `sendBleJson(...)`
- `sendBleJsonInternal(...)`
- `bleGattServer.sendJson(...)`
- `bleGattServer.sendBytes(...)`
- `gattServer.notifyCharacteristicChanged(...)`

发送前置条件：

1. `gattServer != null`
2. `connectedDevice != null`
3. `notifyCharacteristic != null`
4. `notificationsEnabled == true`

## 4.3 分包协议（自定义）

`BleGattServer` 里实现了轻量分包头：

- 1 字节类型：`TYPE_JSON = 0x01`
- 4 字节序号：`seq`
- 包体最大：`510 - 5 = 505` 字节

发送时：

- 小消息单包 `seq=0`
- 大消息按 seq 递增切包

接收时：

- `IncomingPacketAssembler` 按 seq 重组
- 超时（默认 15s）自动 reset
- 兼容兜底：若分包头不完整，会尝试从字节流中抽取 JSON

---

## 5. 业务协议接入（MainActivity）

BLE 收到消息后回调到：

- `handleContentCenterCtrlMessage(String msg)`

该方法会：

1. 解析 JSON
2. 兼容新旧协议封装格式（重点是 `text_input` 信封 + `data.text` 内层 JSON）
3. 统一提取 `action` / `data`
4. 分发业务处理：`handshake`、`ping`、`get_status`、`device_info`、`open_app`、`camera_capture`、提词器、翻译、Wi‑Fi 同步等
5. 通过 `sendBleJson` / `sendAck` / `sendError` 回包

因此从工程视角看：

- `BleGattServer` 负责“链路层（连接与收发）”
- `MainActivity` 负责“应用层（动作分发与业务回包）”

---

## 6. 连接建立最小闭环（可作为联调清单）

手机端要和该眼镜端建立可通信连接，最少要完成：

1. 扫描到服务 UUID `3f6d8b2e-7f61-4c85-b1d2-6e9c0f4a12d7`
2. 发起 GATT 连接并发现服务
3. 找到 Notify 特征 `a1c4e5f6-9b32-4d8a-8c7f-2b9d4e6f1a23`
4. 向其 CCC(0x2902) 写入 `0x0100` 开启通知
5. 找到 Write 特征 `b7d3c1a9-5e24-4fa1-9d6b-3c8e2f5a7b41`
6. 发送首条业务 JSON（建议先发 `handshake`）
7. 接收眼镜端通知回包（ack/response）

只要第 4 步没完成，眼镜端不会成功下行消息。

---

## 7. 当前实现特点与注意点

1. **角色明确**：眼镜是 Server，被手机连接。
2. **无显式配对流程代码**：当前以 BLE 连接 + GATT 权限为主，没有额外 bond 管理逻辑。
3. **通知开关是关键状态位**：`notificationsEnabled` 决定能否回包。
4. **生命周期清晰**：onCreate 启动，onDestroy 停止，避免资源泄漏。
5. **协议兼容性较强**：MainActivity 同时兼容新旧消息封装。
6. **大包可传输**：有分包重组，支持超出单包 payload 的 JSON 数据。

---

## 8. 一句话总结

本工程蓝牙实现本质是：**眼镜端开启 BLE 广播 + GATT Server，手机连上后通过 Write 上行指令、通过 Notify 下行响应；连接后必须先开启 CCC 通知，才能完成稳定双向通信。**
